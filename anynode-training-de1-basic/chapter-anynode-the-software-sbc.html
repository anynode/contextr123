anynode – The Software SBC

As a session border controller, anynode enables incompatible SIP endpoints to communicate with each other without losing features.

A Session Border Controller is used when the signaling and media connections between two separate VoIP networks need to be established, terminated, and transmitted. This mainly includes security, media conversion, connectivity, and regulatory tasks.

With anynode, you can easily rewrite phone numbers, generate routes, add and integrate LDAP directories, and create secure, encrypted connections.

When setting up anynode, our wizard provides many options for configuring the various scenarios in which anynode can be used – Microsoft Teams, Skype for Business, SIP carriers, SIP gateways, and PBXs, or any combination thereof.

Particularly noteworthy are the advantages that result from the fact that anynode is a pure software solution:

  • immediately available via download (e.g. MS Azure, AWS)
  • quick and easy to configure
  • always up to date with the software update service
  • platform-independent
  • fully capable test version

If you just get a little bit of hands-on experience, you realize that not all SIP is the same. So you will encounter numerous compatibility issues. Sometimes you can’t connect one system directly to another system because they don’t understand each other.

For more basic information about anynode, visit the anynode website.

You can get a first insight into the operation and menu structure of anynode in our video workshop “First Steps”.


Back-to-Back User Agent

anynode works on the principle of a back-to-back user agent. In our diagram, you see the provider on the left and the PBX on the right, in the middle is anynode.

At first glance, the black connection lines look like a single call, namely from the provider to the PBX or vice versa. From a purely technical point of view, however, there are two calls. A call from the provider is terminated in anynode. To the PBX, technically speaking, a completely new call is established. This makes it possible to set the entire signaling of these so-called SIP headers to both sides individually.

This allows both systems, which are not compatible, the ability to communicate with each other. anynode’s flexibility allows you to take information from the left side of a certain header and insert it into a completely different header on the other side. Technically speaking, it is two conversations that are actually independent of each other. Both of these connections are one session in terms of licensing and sessions are counted based on incoming calls.

Another task of a session border controller like anynode is the handling of media channels. In the standard configuration, the media channel is always terminated on both sides and transcoding is performed at this point. So if a PBX works with the G.722 codec and the provider is only compatible with G.711, then anynode transcodes the media data from one format to the other.

Later in our exercises, you will perform transcoding and convert media data from one format to the other.

Performance

How many sessions an anynode can process simultaneously depends not only on the license purchased. The available CPU power also plays a role here.

From the performance data table, you can see that, for example, with an 8-core single socket E5 with a high clock rate, over 1000 simultaneous sessions are possible for simple SIP to SIP sessions with little transcoding. With exclusive transcoding from G.711 to G.729, the value drops to 600 simultaneous sessions.

SIP Registrar

anynode includes a registrar function. With anynode, you can create multiple SIP registrars simultaneously. Any SIP registrar can be used to integrate analog devices, SIP phones, existing PBXs, and gateways into a communications environment.

This is especially useful for companies that want to retain their existing equipment, such as

  • analog telephony and fax machines
  • Telephone systems and gateways
  • simple SIP phones
  • SIP intercoms
  • DECT telephony devices
  • and many more …

The SIP registrar allows simultaneous registration of multiple telephony devices at the same time and calls can be switched using simple routing. The devices can be brought together with user-related call number rewriting capabilities in complex topologies.

Further information on the topic “SIP-Registrar” can be found on the anynode website, in the following document, and in the matching video workshop.


Routing and Dial String Rewriting

anynode offers flexible processing of incoming and outgoing source and destination phone numbers. By rewriting different call number formats, anynode makes different IP devices compatible.

Routing can be done for numbers, prefixes or extensions. The route wizard helps to set up the routing easily. Nodes and dial strings can be defined here for routing decisions. Routes based on conditions such as CPU usage, session usage or memory usage are even possible. Furthermore, anynode can route your calls based on time, making more efficient use of additional business hours. anynode allows route supervision based on an existing source such as a customer database.

Trace Analyzer

The anynode Trace Analyzer is a unified analysis tool for technicians on site up to developers of anynode. It includes a listing of all traced calls, a clear representation of the SIP messages of the individual calls as well as an overview with the details of the RTP flows that are either summarized or displayed separately by direction.

The Trace Analyzer is automatically installed with every installation of anynode.

The “onion principle” of anynode

Our simplified diagram gives you an idea of how anynode works. In this example, you see four nodes: a provider, a PBX, a Microsoft Teams, and an XCAPI.

The onion-skin principle clarifies the adaptation, the “modulation”, in the first, outer shell. Here, the header is set so that the other side understands the language. Depending on the setting, anynode can extract the necessary information from different headers.

The call goes into the second onion skin, the “dial string rewriting”, where rewriting of the call numbers can be done if needed. Then the call goes into the “routing”. In the routing, it is decided where the call should go. For example, a call comes from the provider and it is decided in routing that it should be sent to the IP PBX.

anynode Frontend

The upper area in the anynode frontend has various status indicators.

  • User: This displays the currently active username and whether the user has write permissions.
  • Session Timeout: Displays the remaining time until the session expires when no more activity is registered in the frontend.
  • Committed: The “Committed” display confirms that configurations and entries have been finally accepted.
  • State: This shows the status between the frontend and the kernel, this should always be on “connected”.
  • License active: Indicates whether anynode has an installed license. A demo license is automatically activated during initial installation.
  • Trace: Display whether a trace recording is currently activated
